## Executive Summary

The v0.3.0 migration script (`extension/pgmnemo--0.2.1--0.3.0.sql`) is **broken and cannot be applied to a v0.2.1 database.** It references a column `edge_type` that does not exist in v0.2.1 — the actual column is `relation_type`. Two steps fail at runtime. The schema intent (MAGMA §3 edge taxonomy) is architecturally sound, but the implementation has not been tested against the real v0.2.1 schema. Additionally, MAGMA §4 and §5 are not started. Current state is approximately **20% complete** against the full v0.3.0 MAGMA spec.

## 2. installcheck: Broken — 2 Critical Bugs

DB was not reachable at `localhost:15432` for live test. Static code analysis of the migration against the v0.2.1 schema reveals **2 runtime failures**:

### Bug 1 — S3 backfill references non-existent column (line 68)

**File:** `extension/pgmnemo--0.2.1--0.3.0.sql`, lines 66–74  
**Error at runtime:** `ERROR: column "edge_type" does not exist`

```sql
-- BROKEN: references edge_type, which doesn't exist in v0.2.1
UPDATE pgmnemo.mem_edge
SET edge_kind = CASE
    WHEN edge_type IN ('causal', 'derives_from', 'contradicts') THEN 'causal'::pgmnemo.edge_kind
    ...
WHERE edge_kind IS NULL;
```

In v0.2.1, the column is `relation_type` (see `pgmnemo--0.2.1.sql` line 257), not `edge_type`. The migration was authored against a different schema assumption. Because S3 fails, `edge_kind` remains NULL for all rows.

**Cascade failure:** S4 then fails:  
```sql
ALTER TABLE pgmnemo.mem_edge ALTER COLUMN edge_kind SET NOT NULL;
-- ERROR: column "edge_kind" of relation "mem_edge" contains null values
```

The entire migration aborts at S3 — S4 through S8 never execute.

### Bug 2 — traverse_causal_chain() references non-existent column (lines 392, 408)

**File:** `extension/pgmnemo--0.2.1--0.3.0.sql`, lines 382–411  
**Error at runtime:** `ERROR: column me.edge_type does not exist`

```sql
-- BROKEN in both forward and backward branches:
AND me.edge_type = ANY(relation_types)  -- line 392
AND me.edge_type = ANY(relation_types)  -- line 408
```

Even if S3 were fixed, S8 produces a function that fails on every call. The correct column is `me.relation_type`.

### Minor inconsistency — S6 drops phantom index

`DROP INDEX IF EXISTS pgmnemo.ix_pgmnemo_mem_edge_type_time;` — this index does not exist in v0.2.1. The `IF EXISTS` makes it harmless but indicates the migration was not diffed against the actual v0.2.1 schema.

### Fix required (not applied here — task for remediation)

```sql
-- S3 fix:
WHEN relation_type IN ('causal', 'derives_from', 'contradicts', 'CAUSED_BY') THEN 'causal'::pgmnemo.edge_kind
WHEN relation_type IN ('temporal', 'CO_OCCURRED')                             THEN 'temporal'::pgmnemo.edge_kind
WHEN relation_type IN ('semantic', 'elaborates', 'supersedes', 'DERIVED_FROM') THEN 'semantic'::pgmnemo.edge_kind
WHEN relation_type = 'entity'                                                  THEN 'entity'::pgmnemo.edge_kind
ELSE                                                                                'semantic'::pgmnemo.edge_kind

-- S8 fix (both UNION ALL branches):
AND me.relation_type = ANY(relation_types)
```

Note: v0.2.1 stored uppercase values (`CAUSED_BY`, `CO_OCCURRED`, `DERIVED_FROM`, `SUPERSEDES`) in `relation_type`. The backfill mapping must cover both old uppercase values and the new lowercase convention.

## 5. Versioning Recommendation

| Version label | Criteria | Current state | Verdict |
|---|---|---|---|
| **0.3.0** | MAGMA §3 schema working, installcheck passes, recall_lessons() BFS fix validated | Migration broken (2 bugs), installcheck not passing | **NOT YET** — fix migration first |
| **0.3.0** (after fix) | Same as above + relation_type backfill covers uppercase values | Fixable in ~2h engineering | **Appropriate label** once fixed |
| **0.4.0** | MAGMA §3+§4+§5 complete, LongMemEval hybrid recall@10 > 0.95 | §4/§5 not started | Not warranted yet |
| **1.0.0-beta** | H-1 PASS (done) + H-2–H-5 PASS + RESTORE-C1/C2/C3 live | H-2–H-5 blocked, RESTORE not started | Not warranted |

**Recommendation: Ship 0.3.0 only after the migration bug is fixed.**  
The schema change (edge taxonomy) is genuinely meaningful and appropriately labeled 0.3.0 under semver — it's a backward-compatible new column + ENUM type. Do not inflate to 0.4.0 or hold back to 0.2.2.

**Should v0.3.0 be a major version?** No. The change is additive (new column, new type, new indexes, function fixes). Zero breaking changes for existing callers. Major version (1.0.0) requires production validation H-2 through H-5.
