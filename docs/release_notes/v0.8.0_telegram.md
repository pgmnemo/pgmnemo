<b>pgmnemo v0.8.0</b> — Token-Economy Navigation 🧭

Loading every retrieved chunk into an agent's context is what makes retrieval expensive. v0.8.0 splits recall into a cheap <b>LOCATE</b> and an on-demand <b>EXPAND</b>, so you spend context tokens only on what you actually use — ranked across vector + graph + JSONB in one SQL call, inside your own Postgres.

<b>What's new:</b>

• <code>navigate_locate(query_embedding, query_text, token_budget_chars, jsonb_filter)</code> — returns only IDs + scores, stopping once the cumulative budget is reached. No content loaded. The JSONB predicate is pushed into the candidate scan (uses the existing GIN index).
• <code>navigate_expand(ids, expand_fields, graph_expand_depth, ...)</code> — fetches full text for the IDs you choose, plus optional causal/temporal graph neighbours.
• <b>Pattern:</b> locate within a token budget → expand only the rows you need.
• <code>reembed()</code> / <code>reembed_batch()</code> / <code>recompute_content()</code> — safe in-place updates that coexist with live ingestion (no graph rebuild).
• <code>source_type</code> column for origin classification.

<b>Fixed (important):</b>
Graph-proximity in <code>recall_hybrid</code> / <code>navigate_locate</code> was an additive term that could outweigh the retrieval signal by ~10×, burying a perfect vector+BM25 match beneath merely graph-connected rows — a cold-start problem for new or unconnected lessons. It is now a <b>multiplicative tie-breaker</b>: it only re-orders already-relevant candidates. Regression-guarded — a perfect match now reaches the top-3 with the graph term off <i>and</i> at the default weight, even against a dense connected cluster.
