🚀 <b>pgmnemo v0.6.2 released</b>

<b>RRF Fix-A landed — sparse-safe ranking (Cormack 2009)</b>

Real-DB benchmark on LongMemEval-S (N=500, bge-m3 1024d):
• recall@10: 0.9491 → <b>0.9604 (+1.13 pp)</b>
• paired-t p-value: 0.0166 (significant α=0.05)

This resolves the v0.6.1 RRF deferral. The earlier A-scale variant regressed by −22.44 pp on the same corpus; sparse-safe RRF fixes the root cause — high-cosine answers without BM25 match no longer rank below BM25-matching non-answers on small per-item corpora.

<b>What changed</b>

<code>recall_hybrid()</code> now computes a sparse-safe BM25 rank:
<pre>bm25_rank_sparse = CASE WHEN bm25_score > 0
    THEN RANK() OVER (PARTITION BY (bm25_score > 0) ORDER BY bm25_score DESC)
END</pre>

Items with no BM25 match get sentinel rank <code>n_candidates + 1</code> rather than arbitrary ROW_NUMBER positions. Signature unchanged (8 params).

<b>Known deferred: R1 AmbiguousColumn in recall_lessons / recall_hybrid</b>

Agency production benchmark reported <code>psycopg2.errors.AmbiguousColumn</code> on every production call. Root cause turns out to be PL/pgSQL <code>variable_conflict</code> with the RETURNS TABLE OUT variable — not a simple bare-reference issue. Deferred to v0.6.3 to fix properly rather than ship halfway. Production workaround: clients should catch the exception fail-open until v0.6.3.
