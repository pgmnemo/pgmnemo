## S3 — THE PROBLEM
*Purpose: Frame the industry-known problem in terms the target reader already feels. Outcome-confidence wedge. No builder voice.*

---

**Section label (eyebrow text):**  
Why this exists

**Headline:**  
Every agent team eventually bolts on memory. Most end up with three services they can't query as one.

**Body:**  
Vector database for semantic search. Redis or Postgres JSONB for structured lookups. A graph layer for relationships. Three ingestion pipelines. Three schemas to keep in sync. And when you need "semantically similar to X, but in this project, and only from lessons the agent actually trusted" — that query doesn't exist in any of the three. You have to join them yourself.

The other cost: when memory is split across services, your agents can't know which facts they actually relied on. There's no feedback loop. The system has no mechanism to learn that a fact was wrong, or that another one prevented a $15 mistake.

**Transition line:**  
pgmnemo puts all of this in one query plan, inside the database you already have.

## S4 — HOW IT WORKS
*Purpose: Three-step "aha" — make the developer see that the install is real and the API is sane.*

---

**Section label:**  
How it works

**Headline:**  
One extension. One function to write. One function to recall.

**Step 1 — Install:**
```sql
CREATE EXTENSION pgmnemo CASCADE;
SELECT pgmnemo.init_schema();
```
That's the entire install. No sidecar. No config file. pgmnemo lives in your existing Postgres.

**Step 2 — Write memory:**
```sql
SELECT pgmnemo.ingest(
  role        => 'agent_1',
  content     => 'When synthesizing position docs, fold reviewer critiques in first.',
  commit_sha  => 'abc123'   -- or omit to run gate_strict=''off''
);
```
Zero LLM call. Memory write is a SQL constraint check and an indexed INSERT. Costs: one network round-trip to your database.

**Step 3 — Recall:**
```sql
SELECT * FROM pgmnemo.recall_hybrid(
  query_embedding => $1,   -- your embedding
  query_text      => 'positioning synthesis',
  top_k           => 5
);
-- Returns: lesson content · match_confidence · graph_score · provenance
```
Four retrieval channels (vector, BM25, graph proximity, JSONB filters) in one SQL plan. `EXPLAIN ANALYZE` shows the full plan. Token-budget aware: use `navigate_locate()` to get IDs within a character budget, then `navigate_expand()` for only the content you need.

## S5 — WHY (consolidated moat)
*Purpose: The single section that argues why pgmnemo's architecture is defensible. Collapses all prior redundant moat sections into one. Lead with structure, not adjectives.*

---

**Section label:**  
Why this architecture

**Headline:**  
Four things that only work from inside the database

**Item 1 — Single-plan fusion:**
**Vector + BM25 + graph + metadata. One query plan.**  
pgmnemo fuses four retrieval channels — HNSW vector similarity, BM25 full-text, graph-edge proximity (`mem_edge` traversal), and JSONB predicate pushdown — inside a single SQL query plan. The PostgreSQL optimizer manages the join, filter, and ranking. You call one function. No coordinator. No round-trip between services.

No external RAG service can offer this. By definition, an external service is not inside the query optimizer.

**Item 2 — Zero data egress:**
**Embeddings never leave your Postgres.**  
Retrieval, ranking, and scoring happen on the data where it sits. No API token per query. No GDPR surface outside your perimeter. `pg_dump` backs up your memory the same way it backs up your tables. Logical replication replicates it.

**Item 3 — EXPLAIN-able ranking:**
**Run `EXPLAIN ANALYZE` on any recall query.**  
Every retrieval result is a row in a query plan you can inspect. Ranking is regression-testable with SQL. No black-box scoring. You can write a test that asserts a specific lesson appears in the top 3.

**Item 4 — Outcome-learning without a curator:**
**`reinforce('success')` or `reinforce('failure')` — the memory grades itself.**  
When an agent completes a task, its relevant lessons get a confidence update. High-confidence lessons rank higher in future recalls. Low-confidence lessons sink. No manual curation. The memory self-corrects on live outcomes. `match_confidence [0,1]` is an interpretable quality signal your agent can act on.

## S8 — USE CASES
*Purpose: Three personas, each with a specific "I have this problem, pgmnemo solves it" story. Industrial terms, not builder voice.*

---

**Section label:**  
Who uses it

**Headline:**  
Three deployment patterns

---

**Use Case 1 — Agent developer already on Postgres:**

**Title:** You already run Postgres. Memory should live there.

**Body:** You're building agents on top of your existing Postgres stack. You don't want to provision a vector database, a graph store, and a cache just to give agents context. `CREATE EXTENSION pgmnemo` and your agents recall from the same database they write task results to. No synchronization. No consistency boundary. One backup.

**Who this is:** Backend engineers building agent pipelines, RAG systems, or autonomous workflows on Postgres.

---

**Use Case 2 — Cost-sensitive builder running at scale:**

**Title:** At 10,000 agent writes a day, $0.17/1K adds up.

**Body:** Mem0 and Zep run LLM inference on every write — fact extraction, contradiction detection, graph construction. At moderate scale, the write cost becomes a meaningful line item. pgmnemo's write path is a SQL constraint check. No model call. The cost of memory is essentially your existing Postgres connection.

**Who this is:** Teams running agents at volume — automation pipelines, data-processing fleets, multi-agent systems.

---

**Use Case 3 — Team that needs auditability:**

**Title:** The provenance gate: set it and every memory write is verifiable.

**Body:** Set `gate_strict = 'enforce'` and every call to `pgmnemo.ingest()` must carry a `commit_sha` or `artifact_hash`. No application code can bypass it — the check is at the Postgres constraint layer, not in a middleware that can be misconfigured. The audit trail is your database: durable, tamper-evident, restorable with `pg_dump`. For agents touching compliance-sensitive domains (healthcare, legal, GRC), the gate is the only thing that keeps hallucinated facts from accumulating silently.

**Who this is:** Teams building citation-grounded agents, compliance-adjacent AI, or any system where "where did that memory come from?" needs a real answer.
